# Changelog

All notable changes to Cairn are recorded here. Versions are sequential integer build numbers (one increment per `release.ps1` run).

This Changelog starts at the **major rewrite that replaces the legacy 1.x series.** The rewritten libs ship under the same `Cairn-<Name>-1.0` LibStub MAJORs (`Cairn-Addon-1.0`, `Cairn-DB-1.0`, etc.). The Cairn-Gui family keeps its `-2.0` MAJORs unchanged (`Cairn-Gui-2.0`, `Cairn-Gui-Widgets-Standard-2.0`, etc.) since the v2 family was ported verbatim, not rewritten.

The format is loosely based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

### Added

- **`Cairn-Settings-1.0` Cluster E partial 2 — 3 more of the 44 walked decisions land. MINOR 16 → 17.** Extends the schema-builder path with three additive features that all flow through the existing `controlTypes` registry. **Decision 30** — declarative `requireArguments` shape validation. `Cairn_Settings.controlTypes[type].requireArguments` accepts three requirement forms: a Lua type string (`"number"` / `"string"` / `"table"` etc. — required field of given type), a compound table `{ type = "<type>", optional = true|false }` for optional fields, or a custom predicate `function(value, entry) -> bool, errMsg` for shapes a single type string can't express. Built-in `dropdown` ships `requireArguments = { choices = "table" }` and `range` ships optional numeric `min` / `max` / `step` requirements — the inline if-elseif validation in `validateSchema` for `dropdown.choices` is now superseded by the registry-driven path. Consumer-registered types via `:RegisterControl` validate via the same path with zero special-casing. Failures surface at `:New()` time with a clear message including the entry key, type, field name, and reason. **Decision 29** — sub-settings. New `entry.subSettings = {child1, child2, ...}` array on widget descriptors. Children are full schema entries (same key uniqueness + type-validation + default-required rules apply); same flat key namespace as top-level entries (children are NOT scoped under the parent key — sub-settings is about VISUAL nesting + parent-lock, not data scoping). At render time, after the parent initializer is created, the lib walks `subSettings` and registers each child with `SetParentInitializer(parentInit, modifiableFn)` so the child auto-locks when the parent's value is falsy. Default predicate: parent value truthy. Consumer can override via `entry.subSettingsModifiable = function(get) -> bool` for multi-condition locks (parent on AND sibling X in state Y). Children flatten into `_byKey` so `:Get(childKey)` / `:Set(childKey, value)` / `:OnChange(childKey, fn)` work without a sub-namespace dance. **Decision 37** — mixed layout dispatch. `:New(addonName, db, schema, opts)` now accepts an optional 4th arg; `opts.layout = "vertical"` (default) builds a schema-driven panel via `Settings.RegisterVerticalLayoutCategory` while `opts.layout = "canvas"` registers a canvas-layout category via `Settings.RegisterCanvasLayoutCategory(opts.frame, addonName)` for consumers with hand-laid panel frames. Canvas mode skips the schema-based render walk entirely (the consumer owns the panel frame; they render whatever they want into it) but storage / Get / Set / OnChange / default-seeding all still flow through the schema. Missing `opts.frame` in canvas mode logs a warning and falls back to vertical so a misuse doesn't break the panel entirely. The `__call` proxy (`LibStub("Cairn-Settings-1.0")(name, db, schema, opts)`) forwards the new arg. Decisions 28 (compound controls), 31 (three storage backends), 32 (lib_shared_media_dropdown), 36 (already shipped in Cluster A), 39-44 (Cluster F — orthogonal Blizzard-frame override surface) remain deferred. Smoke at `CairnDemo/Smokes/Cairn-Settings.lua` extended with ~28 new assertions covering controlTypes.requireArguments shape, declarative validation rejection paths, consumer-registered-type validation flow, subSettings default-seeding + parent-child key flattening, subSettings rejection paths, and layout dispatch including the canvas-fallback-to-vertical path.

- **`Cairn-Settings-1.0` Cluster E partial shipped — 5 of the 44 walked decisions land in this rev. MINOR 15 → 16.** Control-type infrastructure + dual-keyed category registry on top of Cluster A's schema additions. **Decision 27** — `Cairn_Settings.controlTypes` metadata-rich registry replaces the flat `SUPPORTED_TYPES` set. Each entry carries `storageOnly` (skip widget creation, set value only — applies to `text` / `color` / `keybind`) and `skipDefault` (skip default-value seeding — applies to `header`). The existing `entry.type ~= "header"` default-seed check now reads from `spec.skipDefault` so future header-like types just register with the flag. Bad-type entries surface with a fast `error()` at `:New` registration time via `isSupportedType()` instead of silently skipping. **Decision 33** — `:ModifiedClickOptions(mustChooseKey)` returns a schema fragment producing a paired set of widgets for modifier-key + clickaction settings (the common `CTRL+CLICK to delete` pattern). The fragment shape is `{ {key="...Modifier", type="dropdown", values={NONE,SHIFT,CTRL,ALT}}, {key="...Action", type="text"} }`; consumer flag controls whether NONE is offered or stripped. Reference: LibDualSpec / LibBlzSettings's same-name helper. **Decision 34** — `:RegisterControl(name, spec)` lib-level registration extension point for consumer-defined control types. Spec accepts `storageOnly` + `skipDefault` + an optional `validator(entry) -> bool, errMsg` called during `registerEntry` so consumer-defined types can reject malformed schema entries before they reach the Blizzard panel. Prevents the existing built-in names from being overwritten (loud error). Pattern reference: LibBlzSettings. **Decision 35** — runtime predicate validation at `registerEntry`: `isVisible` / `canSearch` / `newFeature` must be functions or nils; nil-aware so consumers don't have to default-supply. Catches the common typo `isVisible = true` (forgot the function wrapper) early instead of at panel-build time. **Decision 38** — dual-keyed `registeredCategories` map keyed by both addonName AND subName, plus a new `:OpenToCategory(addonName, subName)` jump method. Consumers can deep-link to a sub-panel without knowing the Blizzard `categoryID` — Cairn-Settings looks up the registered tuple and routes through `Settings.OpenToCategory(id)`. The `:_RegisterCategoryEntry(addonName, instance)` internal hook fires from the main `:New` registration path so all panels created via Cairn-Settings appear in the map automatically. **Decisions 28-32, 36-37, 39-44 and the rest of Cluster B/C/D/F (35 more)** still deferred to future phased builds per the Cairn-Settings build plan. Smoke at `CairnDemo/Smokes/Cairn-Settings.lua` extended with ~18 new assertions covering controlTypes metadata, ModifiedClickOptions shape (with/without NONE), RegisterControl + collision rejection, runtime-predicate validation rejection, dual-keyed registry, and OpenToCategory routing (mocked when `Settings.OpenToCategory` not in scope).

- **`Cairn-DB-1.0` substantial reconciliation — 6 of 9 walked decisions landed. MINOR 14 → 15.** **Decision 1** — 8-bucket partition model added alongside the existing `global`/`profile`. New buckets: `char` (per-character per-realm), `realm`, `class`, `race`, `faction`, `factionrealm`. Identity captured at lib load via `UnitName / GetRealmName / UnitClass / UnitRace / UnitFactionGroup` with sensible placeholder fallbacks for non-WoW environments. Defaults table accepts the new keys; merge happens per-bucket. **Decision 2** — wildcard defaults `["*"]` (one-level) and `["**"]` (recursive) via metatable `__index` on the parent table. First access of a missing key materializes a sub-table populated from the wildcard; the recursive variant installs the wildcard at every depth below. Lib-internal keys (those starting with `_`) are skipped to prevent the wildcard fabricating internal state. Plain keys merge non-destructively per existing behavior. **Decision 4** — `:RegisterMigration(version, fn)` migration framework. Each migration fn runs once per addon-version-step; lib walks `saved.internalVersion+1 → max(registered version)` in order at register time + on every subsequent register call. `pcall`-wrapped so a throwing migration routes to `geterrorhandler` without corrupting state. **Decision 8** — `:RegisterNamespace(MAJOR) -> subDB` returns a fresh sub-DB at `saved.__namespaces[MAJOR]` with its own 8-bucket partition. Sub-DBs share identity with parent but are otherwise isolated (own profile / own migrations / own namespaces). Idempotent on MAJOR. **Decision 9** — `Cairn_DB.MIGRATION_DEFER` sentinel value. Migrations returning it queue at lib-scope for a single PLAYER_LOGIN-deferred retry; second defer on retry errors loudly via `geterrorhandler` (silent infinite-defer loops are exactly the failure mode the sentinel exists to prevent). Lazy PLAYER_LOGIN listener installed only when at least one migration defers. Decisions 3 (removeDefaults on PLAYER_LOGOUT) deferred — adds CPU + complexity without immediate consumer demand. Decisions 5 + 6 (compress mode + store-style auto-close) stay future-scoped per the original walk (no consumer demand). Decision 7 was NOTE-ONLY per the walk. **Surfaced Cairn-Settings deps now satisfied:** Decision 19 (RegisterNamespace) and Decision 24 (MIGRATION_DEFER) of the Cairn-Settings walk are unblocked for Cluster D build. The existing smoke assertion checking `New` rejects `realm` as an unknown defaults key updated to use a clearly-bogus key (`realm` is now a valid bucket). Smoke at `CairnDemo/Smokes/Cairn-DB.lua` extended with ~30 new assertions covering all 6 shipped decisions including bucket seeding, wildcard one-level + recursive, migration walk + idempotency + bump, MIGRATION_DEFER queue behavior + version preservation, namespace isolation + idempotency.

- **`Cairn-Timer-1.0` `:ContinueAfterCombat` + 4-mode `:Start` shipped (Decisions 5 + 6 from the 2026-05-12 walk). MINOR 14 → 15.** **`:ContinueAfterCombat(fn)`** — universal "I need to do this but combat lockdown is preventing it" deferral. If `not InCombatLockdown()`, fires the handler synchronously via `Pcall.Call` and returns. Otherwise appends to a lib-scope queue + lazy-installs a `PLAYER_REGEN_ENABLED` listener. On combat-end the queue drains in order via snapshot-then-wipe (so handlers that re-call `:ContinueAfterCombat` don't double-queue into the current drain), each handler `pcall`-wrapped so a throwing handler routes to `geterrorhandler` and doesn't stop the drain. ONE event frame shared across all consumers. Surfaced by Cairn-Settings Decision 12 (consolidated EditModeExpanded's internal CombatManager). **`:Start(slot, mode, period, fn)`** — unified API generalizing four canonical debounce/throttle patterns: **`push`** (trailing-edge debounce; delegates to existing `:Debounce`), **`ignore`** (leading-edge throttle; no-op if slot is already running), **`duplicate`** (fire-and-forget; always schedules a new `:After` regardless of slot state), **`cooldown`** (leading-edge with trailing-merge; fire immediately + merge subsequent calls during the cooldown window into a single trailing fire). Slot state tracked in `Cairn_Timer._slotState[slot]`. Pattern reference: Gopher (Tammya-MoonGuard, 2018). Decisions 1 (string-method support), 2 (fps-drift compensation), 4 (argsCount unpack), 7 (caller-id `from`) DEFERRED — each requires threading through the existing `newHandle` factory which is a deeper refactor that doesn't fit a single-session scope; ContinueAfterCombat + :Start are the highest-impact pieces (the former is a Cairn-Settings dep; the latter unblocks consumer ergonomics for the most common debounce/throttle patterns). Smoke at `CairnDemo/Smokes/Cairn-Timer.lua` extended with ~20 new assertions covering ContinueAfterCombat synchronous path + bad-input rejection, and :Start's four modes including cooldown's pendingTrailing semantic.

- **`Cairn-Log-1.0` reconciliation pass — all 9 walked decisions landed. MINOR 14 → 15.** Substantial architectural changes layered on top of the existing ring-buffer + chat-echo lib, all backward-compat with Forge_Logs and other pre-MINOR-15 consumers. **Decision 1** — Python-style 6-level scheme with gaps: `TRACE=0, DEBUG=10, INFO=20, WARNING=30, ERROR=40, FATAL=50`. `WARN` is preserved as a numeric alias for `WARNING` (both rank 30) so existing call sites (`log:Warn(...)`, `:SetChatEchoLevel("WARN")`) keep working. Gaps reserve room for consumer-defined intermediate levels (VERBOSE=5, NOTICE=25, etc.). Per-level color codes for the new TRACE / FATAL levels too. **Decision 2** — compact `{t, s, m}` entry-shape aliases via metatable `__index` on every pushed entry. The full `{timestamp, source, category, level, message}` shape is unchanged; `entry.t` / `entry.s` / `entry.m` route to the same underlying values for Cluster-A renderers expecting the walked compact form. Both shapes work; no consumer migration needed. **Decision 3** — opt-in database backing via `:SetDatabase(svTable)` lib-level + optional `db` arg on `:New(name, db?)`. When set, every entry that lands in the ring buffer also appends to the consumer-supplied SV table, bounded by the same capacity to prevent unbounded SV growth (oldest entries trim out via `table.remove`). The in-memory ring buffer is unaffected by connect/disconnect; the database is purely additive persistence. `:New(name, db)` bootstraps the lib-level database when none is currently set. **Decision 4** — `pcall` around `string.format` already in `formatMessage`; bad format strings land as `<raw fmt> [format-error: ...]` log entries instead of throwing. **Decision 5** — `:ForceError(fmt, ...)` and `:ForceFatal(fmt, ...)` bypass the chat-echo threshold AND the absent-echo state to guarantee the message reaches the user. Entry still lands in the ring buffer same as a normal call. **Decision 7** — `:SetPerformanceMode(threshold)` nils method slots below the threshold on the shared `LoggerMethods` dispatch table. Consumer gates hot calls via `if Cairn.Log.hasDebug then myLog:Debug(expensive_format(...)) end` — expensive argument construction skipped entirely when DEBUG is disabled. Pass `nil` to restore all methods; originals captured at lib-load so restore is exact. Lib-level scope: affects every logger sharing the metatable; per-instance performance mode is future work (would need per-instance method tables). **Decision 8** — companion `Cairn.Log.hasTrace` / `hasDebug` / `hasInfo` / `hasWarning` / `hasError` / `hasFatal` boolean flags maintained alongside method slots; flip in lockstep with `:SetPerformanceMode`. **Decision 9** — `:Embed(target, name)` mixin injects `:Info`, `:Debug`, `:Warn`, `:Error`, `:Category` onto target so consumers get `self:Info(...)` instead of `self.log:Info(...)`. The `:Log` method is INTENTIONALLY NOT embedded (the key `Log` is reserved for Cairn-Addon's fallback path); consumers wanting custom levels reach through `Cairn.Log:Get(name):Log(...)` directly. Smoke at `CairnDemo/Smokes/Cairn-Log.lua` extended with ~35 new assertions covering levels / entry aliases / database routing / disconnect / Force methods / performance mode threshold / hasX flags / Embed end-to-end.

- **`Cairn-Hooks-1.0` HookOnce family shipped (Decision 5 from the 2026-05-12 walk). MINOR 14 → 15.** Three new methods alongside the existing Pre / Post / Wrap surface — different conceptual model, solves a distinct problem. **`:HookOnce(frame, script, callback)`** installs ONE real `frame:HookScript(script, ...)` on first call; subsequent calls just append to a per-(target, script) callback list. On fire, the entire callback list runs, then the list is WIPED — true "fire once across all subscribers" semantics. Solves the recurring problem of multiple sub-modules each wanting to react to the same frame event firing exactly once. **`:HookAlways(frame, script, callback)`** is the multi-fire variant — same single-real-hook fan-out, but the callback list persists indefinitely. Use when N sub-modules want to react to every fire of a script without stacking N real HookScript installs. **`:HookFuncOnce(table, methodName, callback)`** is the `hooksecurefunc` analogue — installs ONE real `hooksecurefunc(table, methodName, ...)` on first call; subsequent appends share that single install. Fire-once semantics. No `:HookFuncAlways` — `hooksecurefunc` itself supports multiple handlers natively, so the single-real-hook-with-fan-out optimization isn't needed for the persistent variant; use vanilla `hooksecurefunc` for that. Decisions 1-3 (secure-distinction + `:RawHook` + `:FailsafeHook`) DEFERRED — those describe an AceHook-style `:Hook` / `:SecureHook` API that's a different conceptual model from the existing Pre / Post / Wrap surface and needs broader reconciliation work that doesn't fit a single-session scope. Pattern reference: EditModeExpanded's HookOnce.lua (inspected 2026-05-11). Smoke at `CairnDemo/Smokes/Cairn-Hooks.lua` extended with ~12 new assertions covering once-semantics + always-semantics + multi-subscriber fan-out + HookFuncOnce + bad-input rejection.

### Fixed

- **`Cairn-Media-1.0` `:GetQualityColor(name)` strips `|cff` prefix from `ITEM_QUALITY_COLORS[i].hex`.** Blizzard's `.hex` field includes the color-escape prefix (10 chars total: `|cffAARRGG`). Returning that verbatim made consumer code using `Cairn.Util.String.Colorize(text, hex)` produce double-prefixed `|c|cff...|r` malformed strings. Now returns the raw 8-char AARRGGBB form matching `:GetExpansionColor` / `:GetFactionColor` / `:GetThresholdColorHex` conventions. Caught by build 19 smoke; fixed in build 20. Smoke for the below-range threshold-clamp boundary case also fixed (was using ascending stops where below-range = WORST not best; corrected to descending stops where the assertion makes sense).

### Added

- **`Cairn-Media-1.0` color helpers shipped (Decisions 1-3, 5 from the 2026-05-12 walk). MINOR 14 → 15.** Four new methods on top of the existing dual-bucket asset registry: **`:GetExpansionColor(name)`** (Decision 1) — case-insensitive name lookup against a 10-entry private palette covering Classic through Dragonflight, returns 8-char AARRGGBB hex or nil. Recent expansions (TWW / Midnight) flagged as TBD pending community-canonical hex confirmation; absent entries return nil so consumers can default-color or skip. **`:GetQualityColor(name)`** (Decision 2) — thin name-based wrapper around WoW's `ITEM_QUALITY_COLORS` globals. Maps "Poor" / "Common" / "Uncommon" / "Rare" / "Epic" / "Legendary" / "Artifact" / "Heirloom" / "WowToken" (case-insensitive) to the Blizzard-global `.hex` value. Returns nil for unknown names. Cairn-Media doesn't hardcode hex values — Blizzard's globals stay the source of truth. **`:GetFactionColor(name)`** (Decision 3) — same shape as GetQualityColor but for `FACTION_BAR_COLORS`. Reads `.r/.g/.b` floats and formats to AARRGGBB. Includes hardcoded fallback hexes (`FF4A54E8` Alliance, `FFE50D12` Horde) for Classic Era clients where the global isn't present so the lookup never silently fails on a known faction. **`:GetThresholdColor(value, stops...)` + `:GetThresholdColorHex(value, stops...)`** (Decision 5) — multi-stop value-to-color gradient interpolator on a green-yellow-red gradient. Variadic threshold stops accept ascending OR descending order. NaN guard via `value ~= value` (Lua's canonical NaN test — only NaN doesn't equal itself), Inf guard via `math.huge` check, both fall to neutral yellow. Out-of-range values clamp to the nearest stop. Consumers: Forge_AddonManager CPU/memory column coloring, Vellum step-completion display, any quality-bar widget. Decision 4 (Colorize utilities) was scoped to Cairn-Util.String per the walk and shipped in the earlier small-wins batch. - **`Cairn-Slash-1.0` `:GetArgs(str, numargs)` shipped (Cairn-Slash Decision 2). MINOR 14 → 15.** Lib-level helper that splits a string into `numargs` args plus a remaining tail, preserving three non-trivial cases that naive whitespace-split corrupts: quoted strings (`"Cathedral Square"` → one arg), Blizzard hyperlinks (`|H...|h[Display]|h` — item / quest / achievement / spell / encounter / glyph / etc., all sharing the same envelope), and texture escapes (`|T...|t`). Returns `(arg1, arg2, ..., argN, remaining)` where missing args are nil and the remaining tail has leading whitespace trimmed. Handles unmatched quotes (consumes the rest as the token) and malformed hyperlinks/textures (whitespace-split fallback). Decision 1 (weak commands with OnEmbedDisable) and Decision 3 (locale-fallback Sub matching) DEFERRED — both require architectural changes to the existing nested-subcommand-tree model that don't fit a single-session scope. Smoke at `CairnDemo/Smokes/Cairn-Slash.lua` extended with ~16 new assertions covering plain split, quoted strings, hyperlinks, texture escapes, leading/trailing whitespace, more-numargs-than-tokens, unmatched-quote fallback, and bad-input rejection.

- **`Cairn-Util.Hash.Combine` was order-dependent and didn't dedup.** The N-arg fold called `bit.bxor(result, select(i, ...))`, and bare `select(i, ...)` in a function-call position expands to ALL remaining args (`bit.bxor(result, args[i], args[i+1], ..., args[n])`). - **`Cairn-Util.Table.MergeDefaults` clobbered user scalars when defaults supplied a table at the same key.** Previous logic: `if type(v) == "table" then if type(target[k]) ~= "table" then target[k] = {} end; recurse`. - **`Cairn-Flow.FSM` sub-namespace shipped.** Third of four. Hierarchical state machine (HSM, not flat FSM) — composite states declare an `initial` child and their own `states` subtree; the machine is always "in" one leaf and that leaf plus every ancestor are simultaneously active. Public API: `Cairn.Flow.FSM:New({ initial = "X", states = {...} })`, `:Send(event, ...)`, `:GetState()` returns leaf name, `:GetPath()` returns root-to-leaf array, `:IsIn(name)` true for leaf or any ancestor, `:OnTransition(fn)` returns unsub closure. **Event dispatch** bubbles leaf-to-root: each state's `on[event]` is checked in turn; first match wins. - **`Cairn-Util.Numbers` sub-namespace.** Two stateless helpers for human-readable number display. `Numbers.FormatWithCommas(num)` iterates the anchored gsub `^(-?%d+)(%d%d%d)` to fixed point; operates on `tostring(num)` so fractional parts pass through untouched (`1234.5678` → `"1,234.5678"`). `Numbers.FormatWithCommasToThousands(num)` branches on `math.abs(num)`: `< 1000` raw, `< 1M` as `%.2fK`, `>= 1M` as `%.2fM` with commas inserted into the integer part for very large values (`1,500,000,000` → `"1,500.00M"`). No B/T variants in v1 — add when a named consumer asks. Negatives format correctly (`-1500000` → `"-1.50M"`). Known cosmetic quirk: values just under 1M (like 999999) render as `"1000.00K"` because `%.2f` rounds 999.999 up; technically correct but odd. Lives in the consolidated `Cairn-Util.lua`. Cairn-Util MINOR bumps from 20 to 21. Smoke test at `.dev/tests/cairn_util_numbers_smoke.lua`.

- **`Cairn-Util.Path` sub-namespace.** `Path.Get(tbl, "window.position.x")` walks a dot-separated string path through nested tables and returns the terminal value or nil for any missing or non-table intermediate. `Path.Set(tbl, "window.position.x", 100)` does the same walk, auto-creating intermediate tables along the way, and sets the terminal key. Non-table collisions on Set raise an error rather than silently overwriting (`Path.Set({a=5}, "a.b", 10)` errors instead of replacing `tbl.a=5` with `tbl.a={b=10}` — loud failure beats silent data loss). Cairn-Util MINOR bumps from 19 to 20. Smoke test at `.dev/tests/cairn_util_path_smoke.lua`.

- **`Cairn-Util.Hash.FNV1a32` and `Hash.Combine` added.** `Hash.FNV1a32(value, seed?)` is a fast non-cryptographic 32-bit hash via offset basis 2166136261, prime 16777619, per-byte XOR-then-multiply, modulo 2^32. Optional seed is differentiated through the `seed * 13` multiplier so distinct small seeds produce distinct hash spaces even for short inputs. Native `a * b` would lose precision past 2^53; internal `mul32` helper splits the multiplication via 16-bit halves so every intermediate stays within 2^41 (precision-exact modulo 2^32). Each `bit.bxor` is followed by `% UINT32` to normalize WoW Lua's signed-bxor result to the unsigned representation. `Hash.Combine(...)` XOR-folds N hashes with uint32 clamp; order-independent and dup-removing (`Combine(x, x) == 0`) — intentional for anti-entropy bucket aggregation where a bucket represents a SET, not a multiset. Cairn-Util MINOR bumps from 17 to 18.
